## Known Issues & Limitations

### Type Safety
- 50 files use `@ts-nocheck` for type checking bypass
- Type safety improvements planned for v1.1.0+
- See `docs/development/type-safety-roadmap.md`

### Logging
- Console-based logging in production
- Structured logging planned for v1.1.0
- See `docs/development/logging-strategy.md`

### Features
- Calendar functionality is internal (Cal.com deprecated)
- Mobile app not available (web-responsive only)
- API v1 is in beta (breaking changes possible)
- Some features require manual configuration

### Performance
- Database indexes may need optimization under load
- Connection pooling configured but may need tuning
- See `docs/optimization/README.md`

## Rollback Plan

In case of critical issues:

1. **Immediate**: Revert to previous stable version
2. **Database**: Restore from backup if needed
3. **Communication**: Notify users of the issue
4. **Fix**: Address the critical issue
5. **Redeploy**: Deploy fixed version
6. **Verify**: Confirm issue is resolved
7. **Post-mortem**: Document what went wrong

## Release Commands

### Pre-Release

```bash
# Update dependencies
pnpm update

# Run linter
pnpm lint

# Build for production
pnpm build

# Test production build locally
pnpm start
```

### Release

```bash
# Tag the release
git tag -a v1.0.0 -m "Release v1.0.0"
git push origin v1.0.0

# Deploy to production (Vercel)
vercel --prod
```

### Post-Release

```bash
# Monitor logs
vercel logs

# Check deployment status
vercel inspect

# Rollback if needed
vercel rollback
```
